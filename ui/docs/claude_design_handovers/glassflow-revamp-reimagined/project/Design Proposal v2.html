Design proposal · v2

From wizard to control plane: a streamlined plan for the next GlassFlow UI.

We already have hi-fi designs for Library, Canvas, AI, Bridge, and Observability. The new product brief raises the ambition: a real control plane for ClickHouse-oriented streaming — with a dashboard spine, DLQ as a hero workflow, schema evolution, multi-path creation, and proper versioning. This document maps the brief onto what exists, calls out the gaps, and turns each gap into a concrete design task.

01Vision diff

The brief reframes GlassFlow from "a wizard that generates JSON" to "a control plane for streaming pipelines." The current production app is the wizard. Our existing redesign is a strong leap toward the control plane — but it stops short of the full operational story. This is what the diff actually looks like.

Today, in production Built

Step-by-step pipeline wizard (Kafka / OTLP), filter + dedup + ClickHouse mapping, simple pipelines list with health column, import-config dialog. That's it. No reuse, no versioning, no DLQ surface, no schema evolution, no observability beyond a status dot, no AI.

Existing redesign (this project) In progress

Six hi-fi modules already designed: Overview (sitemap), Library (connections, schemas, transforms with version timelines), Canvas (graph editor with live validation), AI (per-pipeline scoped chat with tool-call cards), Bridge (pinned-vs-live model with drift + bulk rollout), Observability (per-pipeline metrics + logs, scoped query badge, brushed range pin). The pinned-vs-live contract is the strongest piece — it solves the “Library is useless / Library breaks prod” dilemma.

What the brief adds Targets v2

The brief pushes for five things our existing designs under-serve: (1) a real Dashboard as the home surface that answers “what needs attention?” (2) a Pipeline-creation landing page with five entry paths instead of dropping users straight into the wizard, (3) DLQ as a product feature — not a side-effect — with replay, suggested fixes, and a path back to schema/filter, (4) Schema evolution as a hero workflow with downstream impact and ALTER TABLE recommendations, (5) Versioning as a first-class concept across pipelines (Draft → Validate → Deploy → Running, with rollback).

The shared internal model. Both the brief and our existing handoff converge on the same atom: Pipeline Draft → Validated → Deployed Version → Runtime State. Every creation surface (wizard, template, canvas, AI, import) produces a draft. The control plane manages drafts and versions. We should pin this model in the spec and never let a surface bypass it.

02Product spine & revised IA

The existing left-rail had four sections (Pipelines, Library, Workspace, Account). The brief argues for a flatter spine where Dashboard is the front door, Pipelines is the runtime view, Library is the asset view, and creation is its own first-class entry. This is the merged tree — green = exists, orange = new in this proposal.

What changed vs the existing IA

  • Dashboard becomes top-level — it was implicit (the “Pipelines list” doubled as home). The brief is right: a real home surface is what makes the product feel like a control plane, not a CRUD app.
  • Create is promoted to a route instead of a primary-button dialog. It hosts the five paths and is where the brief's “don't force one creation flow” gets honored.
  • Library expands to the brief's full resource model: dedup, filters, joins, templates — not just connections / schemas / transforms. These were missing as first-class assets.
  • DLQ and Versions become tabs on Pipeline Detail. They were implied features but didn't have surfaces.
  • Schema evolution gets its own top-level surface. Today, drift only surfaces inside one pipeline; a streaming team needs a queue across all pipelines.

04New surfaces to design

Five new surfaces fill the gaps. Each has a clear purpose, a primary user question it answers, and a defined set of components — so we can scope and build them without scope-creep into “a whole new design system.”

S1

Dashboard / Home

Replaces the pipelines list as the front door. The brief is explicit: this is the “modern infrastructure cockpit” surface.
→ “What needs my attention right now?”
  • System overview row: active pipelines · events/s · error rate · total DLQ events · degraded count
  • Attention queue: schema mismatches, growing DLQs, failed deploys, ClickHouse insert errors — each with a one-click fix path
  • Pipeline status table: name · source → destination · status · throughput · lag · DLQ · last deploy · actions
  • Empty state for first-run: “Create your first pipeline” that links to the new Create surface
S2

Create — pipeline creation landing

A single page that owns all five creation paths. Replaces the “Create” primary button dropping straight into the old wizard.
→ “How do I want to start — click-through, copy a pattern, draw it, describe it, or import it?”
  • Five large entry cards: Guided · From template · Canvas · AI-assisted · Import config
  • Each card shows a one-liner, an icon, and a hint of what comes next
  • All five paths converge to the same draft state, then to Canvas for review/deploy
  • Recently used templates surfaced inline below the cards
S3

DLQ — failed-events workflow

DLQ is hero territory in the brief: turn it from a side-effect counter into the place where users actually recover. Most competitors stop at “here's a count.”
→ “Why did these events fail and how do I fix them?”
  • Failed-event list grouped by reason, with first-seen / last-seen / count
  • Sample event payload viewer with structured field highlight
  • Suggested fix card: cast user_id to UInt64, add filter rule, publish schema v5
  • Replay action with batch-size and rate controls
  • Cross-cuts: “Create filter from error” → Library/Filters · “Update schema” → Schema editor
S4

Versions — pipeline timeline

The brief insists on visible versioning. Today it's invisible: revisions exist in the deploy bar but there's no way to browse, diff, or rollback.
→ “What changed, when, by whom — and can I roll back?”
  • Version timeline: v1 · v2 · v3 · … · current with author + timestamp + change summary
  • Side-by-side config diff between any two versions
  • Deployment status per version (running, paused, failed)
  • Rollback action with a confirm modal that runs the same validation as a fresh deploy
  • Draft state visible if the user has unsaved canvas changes (“Draft v4 exists. Compare → Deploy.”)
S5

Schema evolution — cross-pipeline

Single-pipeline drift exists. The brief asks for a cross-cutting view because schema changes typically affect several pipelines and the ClickHouse target tables.
→ “A field changed at the source — which pipelines and ClickHouse tables are affected, and what should I do?”
  • Detected-change feed (new field, type drift, removed field) per source
  • Impact panel: affected pipelines · affected ClickHouse tables · recommended ALTER TABLE
  • Per-impact actions: ignore · update pipeline schema · update ClickHouse table · pause pipeline · deploy migration
  • Connects back to the Bridge upgrade flow when migration is the chosen path
S6

Library — expanded resources

Library exists. We need three more resource types and a templates index, all reusing the existing list-detail-version pattern.
→ “What reusable building blocks does my team have?”
  • Dedup configs: dedup keys · time window · strategy — pinned, versioned
  • Filters: rule-builder output saved as a named asset — pinned, versioned
  • Joins: temporal join definitions (left/right stream, key, window) — pinned, versioned
  • Templates: ready-to-clone pipeline patterns (Kafka topic → ClickHouse · OTLP logs · dedupe-by-id · …)

05Design principles to hold ourselves to

These are the load-bearing rules — anchored from both the brief and our existing handoff. If a screen violates one, it goes back. (We can argue about which screens; the rules don't move.)

P1
Every abstraction maps to real config

UI concepts must have a 1:1 with what the API actually models. No invented “visual” concepts that confuse what's running.

P2
Every magic action has a review step

AI generates drafts. Templates create drafts. Canvas writes drafts. Nothing deploys without an explicit review of generated config + diff + validate.

P3
Every runtime problem suggests a next action

“Pipeline failed” is a bug. “Pipeline failed because field X expected UInt64, got String — cast it / change ClickHouse / send to DLQ” is the bar.

P4
Canvas stays constrained

Source · Schema · Filter · Transform · Dedup · Join · Sink · DLQ. These are the only nodes. We are not building Node-RED.

P5
Generated config is always visible

Wizard, template, AI, canvas: all four end at a Review screen showing the YAML/JSON, copyable, exportable, diff-able. No hidden mutations.

P6
Defaults are opinionated

40-decision wizards are the failure mode. Recommend sane defaults, expose advanced controls behind progressive disclosure.

P7
Pinned-vs-live is sacred

Connections live · schemas/transforms/dedup/filters/joins pinned. Every surface that lists a Library reference must show its pinned version + drift state.

P8
Pipeline-scoped queries always

Every metrics or logs query carries pipeline_id. The scope badge in the toolbar is the visible enforcement. v1 has no cluster-wide views.

07Open decisions before we build

Things to lock down before Phase A starts — getting these wrong creates rework downstream.

D1 · Where does the user land on login?

Brief says Dashboard. Existing redesign implies Pipelines list. Recommend Dashboard for returning users, but Create / first-pipeline guide for first-run. Confirm with PM.

D2 · Are templates Library assets or a separate concept?

Brief lists Templates as a Library resource. They behave differently from connections/schemas (clone-not-reference). Recommend treating them as Library but with a “clone” verb instead of “pin.”

D3 · Does AI need its own surface or stay drawer-only?

Existing redesign keeps it as a drawer. Brief is open. Recommend drawer for v1, full-page only if usage data justifies it. Avoid scope creep.

D4 · Schema evolution — top-level or under Library?

This is the highest-stakes addition. Recommend top-level because it crosses pipelines + tables + sources. Library is about reusable assets; evolution is about runtime drift.

D5 · DLQ replay — in v1 or stub?

Existing handoff says stub. Brief implies real. Recommend real for v1 because DLQ as “just a counter” doesn't deliver the brief's hero promise. Confirm backend readiness.

D6 · How opinionated should the wizard be?

The brief argues for opinionated defaults. Recommend 3 fields max for happy path in the wizard, everything else under “Advanced.” The current wizard asks for ~12 fields up front; that's the failure mode.
