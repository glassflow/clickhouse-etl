# UI/UX Revamp 2.0 — Master Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement individual phase plans task-by-task. Each phase plan uses checkbox (`- [ ]`) syntax for tracking.

**Goal:** Land the full GlassFlow web app redesign — five product surfaces (Library, Canvas, AI, Bridge, Observability) plus their cross-module wiring — by replacing today's wizard-style UI with a graph-editor + reusable-resource model and adding the M3/M4 observability surfaces.

**Architecture:** App stays on Next.js 16 / Zustand / Drizzle+Postgres. Pipeline detail is restructured from a single page into a tabbed nested route (Overview / Canvas / Library links / Metrics / Logs / Settings). Library gains versioning + detail pages and a new Transforms tab. Canvas extends the existing ReactFlow editor with a palette, validation badges, and a deploy bar. The AI page becomes a global right-side drawer with streaming + tool-call cards. Observability proxies `/ui-api/pipelines/:id/metrics` to VictoriaMetrics and `/ui-api/pipelines/:id/logs` to VictoriaLogs (server-side scope-enforced). Bridge introduces a `pipeline_revisions` + `pipeline_references` table set so schemas/transforms are pinned per revision.

**Tech Stack:** Next.js 16 App Router · TypeScript · Zustand (slice pattern) · Drizzle ORM + PostgreSQL · Tailwind + Radix UI + shadcn primitives · ReactFlow (`@xyflow/react`) · Recharts (new) · Anthropic SDK · Vitest + React Testing Library.

## Critical Architectural Decisions Locked Before Phase 0

These decisions are baked into every downstream phase. Revisiting them mid-build is expensive — flag a deviation explicitly if you must.

### D1 · Pipeline Detail tab structure uses nested App Router routes

The existing `src/app/(shell)/pipelines/[id]/page.tsx` is a single SSR page rendering `PipelineDetailsModule`. We restructure it into:

```
src/app/(shell)/pipelines/[id]/
  layout.tsx          ← shared header + tab nav (NEW)
  page.tsx            ← redirects to /overview (CHANGE)
  overview/page.tsx   ← existing PipelineDetailsModule content (MOVED)
  canvas/page.tsx     ← Phase 2
  library-links/page.tsx   ← Phase 3
  metrics/page.tsx    ← Phase 5
  logs/page.tsx       ← Phase 6
  settings/page.tsx   ← Phase 8 (or stub)
```

Rationale: URL-addressable tabs (sharing a deep link works), each tab can SSR + cache independently, and the layout can hoist the pipeline header + tab nav once. The alternative (in-page tab state) breaks share-links and forces every tab's data fetch into one big component.

This restructure is **task 2.1 in Phase 2** and unblocks tasks in 3, 5, 6, and 8.

### D2 · Bridge uses a `pipeline_revisions` + `pipeline_references` schema

Today, pipelines store config inline. Bridge requires a *pinned snapshot per deploy* so a Library schema bump doesn't auto-mutate any deployed pipeline. New tables (Drizzle, in `ui_library` schema):

```ts
// pipeline_revisions: every Deploy creates one
pipelineRevisions = uiLibrary.table('pipeline_revisions', {
  id: uuid().primaryKey().defaultRandom(),
  pipelineId: text('pipeline_id').notNull(),       // backend pipeline_id
  revision: integer('revision').notNull(),         // monotonic per pipelineId
  config: jsonb('config').notNull(),               // full PipelineConfig snapshot
  createdAt: timestamp().defaultNow().notNull(),
  createdBy: text('created_by'),
})

// pipeline_references: per-revision pin to library resources
pipelineReferences = uiLibrary.table('pipeline_references', {
  id: uuid().primaryKey().defaultRandom(),
  revisionId: uuid('revision_id').references(() => pipelineRevisions.id, { onDelete: 'cascade' }).notNull(),
  resourceKind: text('resource_kind').notNull(),   // 'kafka_connection' | 'clickhouse_connection' | 'schema' | 'transform'
  resourceId: uuid('resource_id').notNull(),
  pinnedVersion: text('pinned_version'),           // null for connections (live)
})

// schema_versions: explicit version history for schemas
schemaVersions = uiLibrary.table('schema_versions', {
  id: uuid().primaryKey().defaultRandom(),
  schemaId: uuid('schema_id').references(() => schemas.id, { onDelete: 'cascade' }).notNull(),
  version: text('version').notNull(),              // semver: '1.4.0'
  fields: jsonb('fields').$type<SchemaField[]>().notNull(),
  changeSummary: text('change_summary'),
  createdAt: timestamp().defaultNow().notNull(),
})

// transforms (new resource type)
transforms = uiLibrary.table('transforms', { /* mirrors schemas */ })
transformVersions = uiLibrary.table('transform_versions', { /* mirrors schemaVersions */ })

// ai_chats (Phase 4)
aiChats = uiLibrary.table('ai_chats', {
  id: uuid().primaryKey().defaultRandom(),
  pipelineId: text('pipeline_id'),  // null = global "new pipeline" chat
  messages: jsonb('messages').notNull(),
  modelId: text('model_id'),
  tokensUsed: integer('tokens_used').default(0),
  updatedAt: timestamp().defaultNow().notNull(),
})
```

These tables are introduced incrementally: `schema_versions` in Phase 1, `transforms` + `transform_versions` in Phase 1, `pipeline_revisions` + `pipeline_references` in Phase 3, `ai_chats` in Phase 4. Each phase ships its migration.

### D3 · Recharts is the chart library

The README permits Recharts, Visx, or hand-rolled SVG. We pick Recharts because:
- Smaller learning curve for the team than Visx
- Built-in brush component (we need this for the Phase 5 brushed-range pin)
- Active maintenance, healthy bundle size with tree-shaking

Install in Phase 0 task 0.5.

### D4 · AI drawer is a portal-mounted overlay, not a route

The AI Assistant must be reachable from anywhere — a route would force unmounting. We mount it via `createPortal` at the root layout, with state in a Zustand `aiUiStore` slice and per-pipeline transcripts hydrated from `/ui-api/ai/chats/:pipelineId`.

### D5 · Observability is server-scope-enforced, not client-scope-enforced

Every `/ui-api/pipelines/:id/metrics?query=…` rewrites the user-supplied PromQL to inject `{pipeline_id="<id>"}`. The UI **never** sends a query that lacks the label — but even if it did, the server rejects unscoped queries. This is the contract that prevented the previous ClickHouse-system-tables observability from working.

### D6 · Wizard is preserved as a first-class alternative — Canvas does NOT replace it

The design handoff README frames Canvas as a *replacement* for the wizard ("Replaces today's wizard-style step-by-step pipeline creation"). **We deliberately diverge from the handoff here.** The existing wizard remains a fully-supported, first-class way to create and edit pipelines. Three independent creation journeys ship side-by-side post-revamp:

| Journey | Entry point | Strength |
|---|---|---|
| **Wizard** (existing) | `/pipelines/create` | Linear step-by-step, validation at each step, lower learning curve, good for first-time users and simple ingest-only pipelines |
| **Canvas** (new, Phase 2) | `/canvas` | Visual graph, multi-source/multi-sink topology, free-form node placement, good for complex pipelines and existing-config edits |
| **AI** (new, Phase 4) | AI drawer (⌘K) | Conversational, generates a draft → opens in Canvas for review |

This decision is locked. **Do not** during any phase:

- Delete or rename `src/app/(shell)/pipelines/create/page.tsx` or its sub-pages
- Delete or refactor `src/modules/{kafka,clickhouse,deduplication,filter,join,otlp,resources,review,transformation}/` as "dead code"
- Delete the wizard-specific store slices or the `StepKeys` constants in `src/config/constants`
- Delete `StandaloneStepRenderer`, `StepRendererModal`, `StepRendererPageComponent`, `PipelineDetailsModule`, or related step renderers under `src/modules/pipelines/[id]/`
- Remove the `coreStore.setTopicCount(n)` / `hydrateFromConfig(config)` flow that the wizard depends on
- Remove the wizard "Create" CTA from the home page

#### What changes for the wizard

These are the **only** wizard-touching changes across the revamp; everything else about the wizard stays exactly as it is today:

1. **Phase 2 (Canvas)** — Pipeline Detail page becomes tabbed (`/pipelines/[id]/{overview,canvas,…}`). The existing `PipelineDetailsModule` (which renders the wizard's step renderers for *editing* a deployed pipeline) **moves** to `/pipelines/[id]/overview`. Its content and behavior do not change. Users editing an existing pipeline still see and use the wizard step UI on the Overview tab.
2. **Phase 0 (Foundations)** — Token alignment may visually refresh wizard surfaces (colors, mono font on IDs/timestamps) but does not alter wizard structure or step flow.
3. **Phase 8 (Polish)** — The empty/loading/error/responsive audit applies to wizard surfaces too, but the same wizard remains.

#### Home page must offer all three creation paths

The home page (`src/components/home/HomePageClient.tsx`) must surface all three journeys — not just one. The existing "Create" CTA flow (sets topicCount → routes to `/pipelines/create`) is the **wizard** entrypoint and stays. Add — alongside it — a "Create with Canvas" CTA that routes to `/canvas`, and the existing/Phase-4 "Create with AI" CTA that opens the drawer with `scope: { kind: 'global' }`. This is part of Phase 4 task 4.8 (already covered) plus a small Phase 2 follow-up:

```tsx
// src/components/home/HomePageClient.tsx — three creation CTAs
<div className="flex flex-col gap-3">
  <Button variant="primary" onClick={handleCreateViaWizard}>Create with guided wizard</Button>
  <Button variant="secondary" asChild><Link href="/canvas">Open canvas</Link></Button>
  <Button variant="ghost" onClick={() => aiUiStore.openDrawer({ kind: 'global' })}>Ask AI</Button>
</div>
```

Phase 2 task 2.9 step 3 must be amended: when updating `/canvas/page.tsx`, **do not** remove the wizard route or its CTA; the canvas page is an *additional* entry point, not a replacement.

#### Pipeline editing — both wizard and canvas work post-deploy

For an already-deployed pipeline, users can edit via either tab on `/pipelines/[id]`:

- **Overview tab** → wizard-style step renderers (existing `PipelineDetailsModule`, no change)
- **Canvas tab** → graph editor (Phase 2)

Both tabs read from and write to the same `coreStore`/domain stores. No content lives only in one tab — switching between them mid-edit must not lose user input. The `coreStore.hydrateFromConfig(config)` flow is the contract that keeps both views in sync; do not bypass it.

#### Acceptance criteria addition (Phase 8)

Add to the Phase 8 self-audit:

- [ ] `/pipelines/create` (wizard) still loads and the full step flow works end-to-end (Kafka source → topic → dedup → ClickHouse sink → review → deploy)
- [ ] Editing a deployed pipeline via Overview tab still uses wizard step renderers
- [ ] Home page surfaces all three creation paths (Wizard / Canvas / AI), each functional
- [ ] No file under `src/modules/{kafka,clickhouse,deduplication,filter,join,otlp,resources,review,transformation}` was deleted by any phase

## Cross-Cutting Constraints (apply to every phase)

These come straight from `CLAUDE.md` and the design handoff. Treat them as compile-time invariants.

1. **No hardcoded colors.** Every `bg-*`, `text-*`, `border-*` Tailwind utility for color is forbidden. Use `var(--token-name)` exclusively. (Layout/spacing Tailwind utilities are fine.)
2. **Use variant props on primitives, not raw class names.** `<Button variant="primary">`, not `className="btn-primary"`. `className=` on primitives is for layout only.
3. **Dark theme only.** No light-mode branches.
4. **Mono font discipline.** `JetBrains Mono` (or system mono fallback) for IDs, timestamps, axis labels, query strings, code. Body sans elsewhere. Don't mix.
5. **Pinned-vs-live contract.** Connections: live edits, blast-radius dialog. Schemas/Transforms: pinned per pipeline revision; upgrade is a new revision. Every UI surface that lists references must reflect this — visually distinct.
6. **Pipeline-scope contract.** Every metrics/logs query is rewritten server-side to include `{pipeline_id="<id>"}`. The `scoped: <id>` badge is the visible affordance.
7. **One AI chat per pipeline + one global "new pipeline" chat.** Drawer header always shows the scope.
8. **Brushed-range global filter.** A range pinned on Metrics propagates to Logs via the observability store; pinned-range pill is dismissable.
9. **Every async surface needs all four states.** Empty (no data yet), loading (skeleton, no spinner where possible), error (scoped per query/card, never blanks the page), populated.
10. **Layout shape is preserved between states.** A loading skeleton occupies the same chart frame as the populated chart. No flicker on state transitions.
11. **Wizard is preserved (D6).** `/pipelines/create` and `src/modules/{kafka,clickhouse,deduplication,filter,join,otlp,resources,review,transformation}` stay. Canvas and AI are *additional* entry points, not replacements. Home page offers all three creation paths.

## Execution Handoff

Each phase plan can be executed independently. Recommended workflow:

1. **Engineer or agent picks one phase file** based on the parallelism table above.
2. **Run that phase end-to-end** in its own branch (`phase-N-slug`) following the bite-sized tasks inside.
3. **Open a PR per phase** — keeps reviewable scope sane and lets phases land in priority order.
4. **Merge order:** Phase 0 first. Then any of {1, 2, 4, 5, 7} in any order. Phase 3 after 1 + 2. Phase 6 after 5. Phase 8 last.

For each phase, the writing-plans skill recommends choosing one of:

- **Subagent-Driven (recommended):** `superpowers:subagent-driven-development` — fresh subagent per task, review between tasks.
- **Inline Execution:** `superpowers:executing-plans` — batch execution with checkpoints.

Pick per-phase based on team preference. Phase 0 is small enough for inline; Phases 2/5/6 are large enough that subagent-driven shines.
