OTLP Source Support — Architectural Approaches

How should we integrate OTLP as an alternative data source in the pipeline wizard?

A

Source-Aware Journey Builder

Recommended — Add sourceType to CoreStore. Home page gains a source selection step before topic count. A new getOtlpJourney() function returns the OTLP-specific step sequence. The wizard, sidebar, and components branch on sourceType.

Pros

  • Follows existing journey pattern — getSingleTopicJourney() / getTwoTopicJourney() / getOtlpJourney()
  • Minimal changes to wizard orchestrator — it already works with dynamic step arrays
  • New OTLP-specific steps (signal type selector, OTLP dedup) are isolated in src/modules/otlp/
  • Reuses existing ClickHouse, filter, transformation, and resources steps
  • Clean separation — Kafka code untouched

Cons

  • Need a new store slice (OtlpSlice) for signal type and OTLP-specific config
  • Home page gets a bit more complex (source selection → then topic count for Kafka only)
  • Some shared steps (dedup, mapper, resources) need sourceType-aware branching
B

Separate OTLP Wizard Route

Create an entirely separate wizard at /pipelines/create/otlp with its own page, step definitions, and components. Home page routes to either /pipelines/create (Kafka) or /pipelines/create/otlp.

Pros

  • Complete isolation — zero risk of breaking existing Kafka flow
  • OTLP wizard can be optimized for its simpler flow
  • Easy to feature-flag or remove independently

Cons

  • Significant code duplication — ClickHouse connection, mapper, resources, filter, transformation steps duplicated
  • Two wizard orchestrators to maintain
  • Divergence risk — shared steps evolve independently, bugs get fixed in one but not the other
  • Store slices still shared, so "isolation" is partial
C

Abstract Source Provider Pattern

Introduce a SourceProvider interface that both Kafka and OTLP implement. Each provider defines its own steps, store slices, schema shape, and validation. The wizard consumes the provider generically.

Pros

  • Most extensible — adding a third source type (e.g., S3, Pub/Sub) follows the same pattern
  • Clean inversion of control — wizard doesn't know about specific sources
  • Shared steps (ClickHouse, resources) remain truly shared

Cons

  • Over-engineering for 2 source types — significant refactor of existing Kafka code to conform to the provider interface
  • High risk — touches many existing files, could break Kafka flow
  • Longer timeline — abstraction design + migration + new OTLP provider
  • YAGNI unless a third source is planned soon
