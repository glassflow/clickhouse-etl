What data density lives at the command center level?

The top-level /observability page shows all pipelines. The question is how much metric signal lives inline vs. only available after you click into a pipeline.

A

Status-first (lightweight)

Each pipeline row shows: name, status badge, DLQ count, last-seen throughput number (a single value). No sparklines. Fast to load, low VM query cost. You drill in to see charts.

Command center — status table
PIPELINE STATUS THROUGHPUT DLQ ERROR RATE
prod-orders ● active 12,450 ev/s 0 0.01%
analytics-stream ● degraded 3,210 ev/s 47 2.4%
user-events ● paused — 0 —

Pros

  • Very fast to load
  • Scales to 50+ pipelines
  • Easy to scan

Cons

  • No trend visibility — can't see "it was fine, then spiked"
  • A single throughput value hides anomalies
B

Sparkline-enhanced (medium density)

Each row has a 60-minute throughput sparkline and a 60-minute error-rate sparkline pulled from VM. Instantly shows trends and anomalies without needing to click in. Moderate VM query cost (one query per pipeline).

Command center — sparkline table
PIPELINE STATUS THROUGHPUT (1h) ERRORS (1h) DLQ
prod-orders ● active 0
analytics-stream ● degraded 47

Pros

  • Trend visibility at a glance — "it was fine, then dropped"
  • Degraded pipelines stand out visually
  • Cross-pipeline throughput comparison is instant

Cons

  • N VM queries for N pipelines (can batch)
  • Slightly slower first paint
  • More complex to implement
C

Card grid (high density)

Each pipeline gets a card with a small chart, status, and key numbers. More visual, works well for 6–12 pipelines. Gets crowded at 20+.
