# GlassFlow ClickHouse ETL UI — Design Context

## Design Context

### Users
Primary: data engineers configuring Kafka→ClickHouse ETL pipelines. Extended technical work sessions, primarily desktop. Comfortable with infra tooling; prioritize clarity, legibility, and precision. Secondary: less-technical stakeholders (data team leads, ops) who monitor pipelines and need to understand system health at a glance without deep CLI knowledge. Design must serve both: power-user density without shutting out the newcomer.

### Brand Personality
**Modern, data-native, clear.** Orange as warmth and energy in a cold enterprise space. Aesthetic anchors: Linear's compositional precision and type hierarchy, Supabase's accessible confidence and readable prose, GlassFlow's own dark-native surface language. Not playful, not enterprise-gray, not terminal-stark. Feels like it belongs next to dbt, Kafka, ClickHouse — not like it was generated by a dashboard template.

### Aesthetic Direction
Dark-mode native (only). Near-monochromatic charcoal elevation stack (#070708 → #0a0a0d → #0c0c10 → #0e0e12 → #161617) with a single amber-orange accent (#ffa24b). Monospace font (JetBrains Mono) for all data labels, metrics, and pipeline identifiers — this is a signature of the data-terminal aesthetic. Gradient CTA button is a brand signature moment. Anti-references: Oracle/SAP gray monotony, purple-to-blue AI palettes, left-edge stripe callouts, rounded-square icon + heading card grids.

### Reference products
- **Linear** — for type hierarchy, interaction precision, and compositional restraint
- **Supabase** — for confident prose, accessible density, and developer-friendly data display  
- **GlassFlow's own design system** — the dashboard aesthetic (deep blacks, orange accents, monospace metrics) is the gold standard to replicate elsewhere

### What to preserve
- Primary color: `--color-orange-300` (#ffa24b) and its gradient (#feac5e → #e28024)
- Dark theme (dark-only, no light mode)
- The gradient CTA button
- The nav tab underline gradient indicator
- The `surface-gradient-border` modal signature
- The dashboard's monospace metric/label pattern — this is the distinctive data-native voice
- Orange alpha hover pattern (`--color-orange-alpha-10` bg + `--color-orange-300` border)

### Known anti-patterns to remove
- **Left-edge colored stripes** (`border-left: 3+px solid <color>`) — used in `.healthy-banner` and `.attn-stripe`. Replace with background tints or full borders.
- **Decorative sparklines** — `Spark` component in KPI cards has no interaction. Either add tooltip-on-hover with real data, or remove.
- **Fade-to-transparent directional gradients** on banners — `linear-gradient(90deg, color 0%, transparent 80%)` feels AI-generated.
- **Raw px font sizes** in dashboard CSS — should align to the type scale tokens.

### Design Principles
1. **Data engineers first** — clarity over decoration; explicit over clever; density is a feature, not a problem
2. **Accessible to non-technical readers** — status, health, and key numbers must be legible without domain knowledge
3. **Premium, not generic** — fight the admin-template look; the dashboard style is the goal, not the starting point
4. **Visual coherence** — dashboard, wizard, library, and observability must feel like one product, not four tools stitched together
5. **Type and spacing discipline** — use the token system consistently; spacing rhythm and type hierarchy are the primary drivers of perceived quality
6. **State through weight, not just color** — incident states use visual weight escalation (size, contrast), not only hue change

### Token system
Full token system in `src/themes/base.css` (primitives) + `src/themes/theme.css` (semantic). Never use raw Tailwind color utilities for semantic/brand colors. All spacing via `--unit-x*` tokens (4px base). Typography via `.title-*`, `.body-*`, `.caption-*` utility classes. Dashboard-specific tokens live under `/* ============ Dashboard Tokens ============ */` in `theme.css`.
