# Avro Pipeline Usage

<Tier badge="enterprise" />

GlassFlow supports [Apache Avro](https://avro.apache.org/) as a Kafka source format. Messages are decoded before entering the processing pipeline, so deduplication, filtering, stateless transformations, and the ClickHouse sink all work the same way as with JSON sources.

<Callout type="info">
Avro (and Protobuf) format support is available in the Enterprise Edition only. JSON sources are available in both Open Source and Enterprise. [Get in touch](https://www.glassflow.dev/integrations#contact) to request access.
</Callout>

## How it works

GlassFlow treats Avro as an **external format**: the Confluent-envelope payload is decoded into a flat field map at ingestion time, and the rest of the pipeline operates on those fields just as it would with a JSON source. There is no change to the internal wire format.

Two schema sources are supported:

| Mode | When to use |
|------|-------------|
| **Inline** | You own the `.avsc` definition and want to embed it directly in the pipeline config. |
| **Confluent Schema Registry** | Your producers register schemas with a Confluent-compatible SR; GlassFlow fetches the schema by the ID embedded in each message. |

In both modes every Kafka message must use the **Confluent wire format**:

```
[0x00 magic byte][4-byte big-endian schema ID][serialized Avro payload]
```

## Inline schema

Provide the `.avsc` schema as a JSON-escaped string under `schema.file`. GlassFlow parses and validates it at pipeline-creation time.

**Constraints for inline schemas:**
- Schema text must be ≤ 64 KB.
- The root type must be a `record`.

## Confluent Schema Registry

Add a `schema_registry` object alongside a `schema` seed. The seed is **required**: GlassFlow stores it as the base schema version at pipeline-creation time. When a message arrives with a different schema ID, GlassFlow fetches the new version from the registry and validates backward compatibility against the stored base.

<Callout type="info">
**Schema evolution.** When a producer registers a new backward-compatible schema version (e.g. adding an optional field), GlassFlow fetches and caches the new schema automatically when the new ID first appears in an envelope — no pipeline restart needed.
</Callout>

## Union and nullable fields

Avro nullable fields use a union of `["null", T]`. GlassFlow resolves the non-null branch and uses its type for field registration:

```json
{ "name": "country", "type": ["null", "string"], "default": null }
```

This field is registered as type `string` and can be mapped to a `String` ClickHouse column. A `null` value at runtime is written as `NULL` if the column is `Nullable(String)`, or as the ClickHouse default otherwise.

Unions with more than two branches (e.g. `["null", "string", "int"]`) resolve to the first non-null branch.

## Nested record fields

Avro nested records are supported. You can map individual sub-fields using dot notation or map the whole nested record as a single `String` column.

```json
{
  "type": "record",
  "name": "Event",
  "fields": [
    { "name": "id", "type": "string" },
    {
      "name": "metadata",
      "type": {
        "type": "record",
        "name": "Metadata",
        "fields": [
          { "name": "source",  "type": "string" },
          { "name": "version", "type": "int"    }
        ]
      }
    }
  ]
}
```

Sink mapping using dot notation:

```json
{ "name": "metadata.source",  "column_name": "metadata_source",  "column_type": "String" },
{ "name": "metadata.version", "column_name": "metadata_version", "column_type": "Int32"  }
```

## Field type mapping

### Primitive types

| Avro type | GlassFlow intermediate type | Supported ClickHouse types |
|-----------|----------------------------|---------------------------|
| `string`  | `string` | String, UUID, FixedString, LowCardinality(String) |
| `int`     | `int32`  | Int32, Int16, Int8 |
| `long`    | `int64`  | Int64, DateTime, DateTime64 |
| `float`   | `float32` | Float32 |
| `double`  | `float64` | Float64, DateTime, DateTime64 |
| `boolean` | `bool`   | Bool |
| `bytes`   | `bytes`  | String |
| `fixed`   | `bytes`  | String |
| `enum`    | `string` | String, LowCardinality(String) |
| `array`   | `array`  | Array(String), Array(Int64), etc. |
| `record`  | `object` | String (as JSON), or map sub-fields with dot notation |

### Logical types

Avro logical types are decoded by GlassFlow to their native Go representation. Map them to the corresponding ClickHouse type:

| Avro logical type | Avro base type | Decoded as | Recommended ClickHouse type |
|-------------------|---------------|------------|-----------------------------|
| `timestamp-millis` | `long` | `time.Time` | `DateTime64(3)` |
| `timestamp-micros` | `long` | `time.Time` | `DateTime64(6)` |
| `date`             | `int`  | `time.Time` | `Date` |
| `time-millis`      | `int`  | `time.Duration` | `Int32` (milliseconds) |
| `time-micros`      | `long` | `time.Duration` | `Int64` (microseconds) |
| `uuid`             | `string` | `string` | `UUID` or `String` |
| `decimal`          | `bytes`/`fixed` | `*big.Rat` | `String` (serialized) |

Example schema using logical types:

```json
{
  "type": "record",
  "name": "Order",
  "fields": [
    { "name": "order_id", "type": "string" },
    {
      "name": "created_at",
      "type": { "type": "long", "logicalType": "timestamp-millis" }
    },
    {
      "name": "event_date",
      "type": { "type": "int", "logicalType": "date" }
    },
    {
      "name": "request_id",
      "type": { "type": "string", "logicalType": "uuid" }
    }
  ]
}
```

Sink mapping:

```json
{ "name": "order_id",   "column_name": "order_id",   "column_type": "String"       },
{ "name": "created_at", "column_name": "created_at", "column_type": "DateTime64(3)" },
{ "name": "event_date", "column_name": "event_date", "column_type": "Date"          },
{ "name": "request_id", "column_name": "request_id", "column_type": "UUID"          }
```

## Source configuration reference

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `format` | string | Yes | Must be `"avro"`. |
| `schema` | object | Yes | Schema definition. For inline mode, set `schema.file` to the `.avsc` text (JSON-escaped string). Required for both inline and Schema Registry modes. |
| `schema_registry` | object | No | Confluent Schema Registry connection. When present, enables registry mode. |

### `schema` fields

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `file` | string | Yes | The `.avsc` schema text as a JSON-escaped string. Required for both inline and Schema Registry modes. |

### `schema_registry` fields

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `url` | string | Yes | Base URL of the Confluent-compatible Schema Registry. |
| `api_key` | string | No | API key for basic authentication. |
| `api_secret` | string | No | API secret for basic authentication. |

## Troubleshooting

### `avro schema parse error`

The `schema.file` string is not valid Avro JSON. Common causes: missing `type` field, unknown named type, or a field referencing a record not defined in the same schema. Check the error message from hamba/avro for the specific location.

### `avro schema root must be a record`

The root `type` in `schema.file` is not `"record"`. Avro schemas for GlassFlow must have a record at the root.

### `avro schema is N bytes, exceeds 65536 byte limit`

The inline `.avsc` JSON exceeds the 64 KB limit. Use a Schema Registry instead.

### `schema.file is required for inline avro`

The `format` is `"avro"`, no `schema_registry` is present, and `schema.file` was not provided. Add the `.avsc` text as a JSON-escaped string under `schema.file`.

### `avro unmarshal: ...`

A message failed to decode at runtime. Possible causes:
- The producer is not using the Confluent wire format (missing `0x00` magic byte).
- The message was encoded with a schema incompatible with the stored version.

Use the [Dead-Letter Queue](/configuration/dlq) to inspect failed messages.

### `schema not found` after evolution

A producer published a message with a new schema ID that is not registered in the Schema Registry, or the registry credentials are incorrect. Check that the Schema Registry URL, `api_key`, and `api_secret` are correct and that the schema is registered.

### Incompatible schema evolution

GlassFlow validates new schema versions against the stored base. Breaking changes (renaming or removing fields, changing field types) cause the pipeline to emit a signal and route the message to the DLQ. Use additive evolution only: add new fields with defaults, do not remove or rename existing ones.
